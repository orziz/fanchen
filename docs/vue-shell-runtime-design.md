# Vue 外壳与源码运行时设计

## 体验目标

- 玩家打开编译产物后，第一眼仍然是完整游戏界面，而不是空白页或技术启动器。
- 启动期间要看得见“正在启动源码运行时”的过程。
- 运行时失败时，页面应给出明确反馈和重试按钮，而不是静默白屏。

## 页面行为

### 启动态

- Vue 先渲染完整壳层 DOM。
- 中央覆盖一层轻量启动卡片，显示当前加载阶段。
- 运行时启动器推进到主循环前，更新阶段文案。

### 就绪态

- 源码运行时成功初始化后，覆盖层消失。
- 后续页面交互全部继续由同一源码树内的运行时逻辑驱动。

### 失败态

- 保留已经渲染好的页面骨架。
- 中央卡片显示失败原因。
- 提供“重新启动”按钮，再次尝试拉起运行时。

## 组件边界

### App.vue

- 只负责启动状态、错误态和总控覆盖层。

### GameShell.vue

- 只负责输出与现有 DOM id 完全对齐的页面骨架。
- 不掺入玩法逻辑。

### useGameRuntime

- 管理 booting / ready / error 三种状态。
- 调用源码运行时启动入口。
- 将底层加载进度翻译为玩家可读文案。

### bootstrap.ts

- 负责静态导入运行时源码模块。
- 负责避免重复启动。
- 负责在 Vue 外壳就绪后手动触发 initialize。

## DOM 对齐规则

- 所有运行时 bindDom 依赖的 id 必须保持不变。
- 所有事件委托依赖的 data 属性必须保持不变。
- 窗口系统依赖的 data-window、data-window-handle、data-window-action 也必须保持不变。

## 设计取舍

### 为什么先保留当前 DOM 结构

- 这能让迁移第一步就交付可玩的编译产物。
- 这能把风险集中在“源码运行时是否稳定启动”，而不是把页面和玩法一起重写。

### 为什么启动覆盖层要独立存在

- 运行时虽然已经收束为源码模块，但主循环、状态恢复和自动存档仍需要一个明确启动反馈。
- 后面即使逐步把系统迁到 TS，启动覆盖层依旧可以复用为资源预热入口。

### 为什么错误态只给重试，不给更多技术细节

- 玩家视角只需要知道能否进入游戏。
- 过多技术细节应留在控制台，而不是占据界面主文案。
