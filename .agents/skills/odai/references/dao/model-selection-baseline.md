# 模型选择基线

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 当前环境已经暴露可分配模型列表，主流程需要给并行席位分配模型
- 需要判断当前暴露的模型是否足以成立完整多模型合议模式
- 需要给结果总结补出“为何没有进入完整多模型合议模式”的模型层原因
- 需要为统一下发包填写“本席位模型”与“本席位模型选择理由”

## 总原则

1. 先判模式是否合法，再谈模型最优分配；不满足完整能力条件时，不得靠“看起来不同”硬判成多模型合议。
2. 主流程优先求稳，子席位优先求异；能异构就异构，但异构要建立在环境已明确暴露模型标识之上。
3. 先按席位职责选模型，再按模型名气或主观偏好选；不要把“强模型”堆到所有席位上做同质化重复劳动。
4. 只允许使用环境已暴露的信息、平台文档、仓库内已验证经验或本轮可直接观察到的表现给模型打标签；不得编造模型特长。
5. 若模型列表未暴露，或只暴露了模糊标签，则回退到“子 agent 辅助复核模式”或原流程，并在结果总结里明说原因。

## 进入选择前必须拿到的输入

1. 当前环境暴露的可分配模型列表。
2. 每个模型可直接读取到的标识：provider、model id、display name 或平台暴露的等价字段。
3. 当前模式目标：完整多模型合议模式，还是子 agent 辅助复核模式。
4. 当前要分配的席位清单与职责。
5. 当前任务的主要约束：稳定性、长上下文、速度、成本、读档规模、是否需要强反例挑战。

若以上信息缺失到无法判断“是不是不同模型”或“该给谁分什么模型”，就不要继续假设，直接降档。

## 模型归类方式

先按已暴露信息和已验证经验，给候选模型打职责标签。一个模型可以同时命中多个标签，但标签必须有根据。

- 综合收束型：适合主流程、方案收束、长链路总结、冲突裁决。
- 题面挑战型：适合找偷换概念、补未确认点、打断默认假设、提反例。
- 风险审计型：适合风险分级、边界校验、遗漏检查、冻结后复查。
- 速扫补证型：适合大量读文件、补目录、补相邻证据、快速查漏。
- 长上下文型：适合需要一次吞入较多材料、长文档或多文件证据的席位。

若环境没有暴露足够信息来可靠打标签，只保留最保守标签：`可用但特长未验证`。

## 席位分配基线

### 主流程

- 优先选择综合收束型。
- 若多个模型都能胜任，优先选已验证更稳、长上下文更充足、总结偏差更少的那个。
- 主流程模型一旦确定，完整多模型合议模式下的首个子席位应尽量换成不同模型。

### 确认前题面校验席位

- 优先选择不同于主流程的题面挑战型或风险审计型。
- 目标不是重复主流程总结，而是找偷换意图、范围漂移、证据缺口与错误前提。

### 确认后首轮合议席位

- 至少保留一个反例挑战席位。
- 若任务偏实现或审查，可加一个风险审计型或速扫补证型席位。
- 不要让两个席位同时使用同一模型做同一职责。

### 冻结后独立复查席位

- 能换模型就换模型；能换职责视角也要换。
- 若只能继续使用同一模型，必须至少更换席位职责，并在结果总结里说明未能做到模型异构。

## 不同模型的判定口径

1. 只有当平台明确暴露出可单独分配的不同模型标识时，才可判定为“不同模型”。
2. 同一模型的不同 system prompt、温度、推理档位、输出风格或角色设定，不算不同模型。
3. 同一基础模型的包装别名、路由别名或仅展示名不同，不算不同模型。
4. 若平台明确把不同版本、不同尺寸或不同发布日期的模型列成可分配的独立模型，可以视作不同模型；但完整多模型合议仍优先选择跨家族或跨能力侧重点的组合，而不是只换一个轻微变体。
5. 只要无法确认是不是不同模型，就按“不能确认”处理，不得为了开完整模式而乐观放行。

## 完整多模型合议模式的分配顺序

1. 先选主流程模型。
2. 再选至少 1 个不同模型做题面挑战或反例挑战席位。
3. 若需要第 3 个席位，优先补风险审计、冻结后复查或速扫补证，而不是复制已有职责。
4. 每增加一个席位，都要先回答三个问题：
   - 这个席位解决的是哪类剩余不确定？
   - 为什么现有席位还不够？
   - 为什么这个模型比现有模型更适合这个席位？

若这三个问题答不清，就不要继续扩席位。

## 子 agent 辅助复核模式的分配顺序

1. 默认只拉 1 个子 agent。
2. 若环境暴露了模型列表，且存在明显更适合做反例挑战或风险审计的不同模型，可以优先选那个模型。
3. 若模型列表未暴露，或只暴露单一模型，则仍可进入辅助复核模式，但结果总结里必须写清：当前环境未暴露模型列表，或当前仅暴露单一模型，因此未进入完整多模型合议模式。
4. 只有当第 1 个子 agent 返回后仍存在真实主路分歧或高风险未收束，才考虑追加第 2 个子 agent。

## 选择步骤

1. 先把环境暴露的模型标识标准化成一组稳定字段。
2. 去掉明显只是同模型不同预设、不同标签或不同角色包装的重复项。
3. 判断当前可确认的不同模型数量。
4. 按席位职责给候选模型打标签；没有把握就写“特长未验证”。
5. 先分配主流程，再分配挑战席位，再看是否需要补审计或复查席位。
6. 把每个席位的模型与选择理由写进统一下发包。
7. 若任一步骤拿不准，就降档，不为追求形式保留完整多模型合议模式。

## 结果总结必须回报的模型层信息

- 当前进入的是完整多模型合议模式，还是子 agent 辅助复核模式。
- 当前环境是否暴露了可分配模型列表。
- 本轮实际调用了哪些模型；若环境未暴露，则明确写“当前环境未暴露”。
- 若没有进入完整多模型合议模式，模型层面的直接原因是什么：
  - 模型列表未暴露
  - 仅暴露单一模型
  - 无法确认至少两种不同模型
  - 模型虽不同，但没有足够依据做可靠席位分配

## 短规则

- 先判合法，再判优选。
- 先定主流程，再配挑战席位。
- 能异构就异构，不能异构就诚实降档。
- 不靠别名、角色和推理档位伪造不同模型。
- 模型选择理由必须写入统一下发包。
